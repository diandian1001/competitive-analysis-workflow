# Benchmark：AI 竞品与竞争情报操作系统

> 状态：测试设计完成，待实际运行。未运行前不得声称 V4.0 已被实证证明优于普通 AI 对话。

## 一、目标

验证 Workflow 是否可以稳定提升：

- 业务决策定义；
- 我方基线完整性；
- 竞品选择合理性；
- 事实准确性；
- 证据可追踪性；
- 机会判断质量；
- 行动建议可执行性；
- 跨模型稳定性。

## 二、实验设计

每个任务设置：

| 组别 | 条件 |
|---|---|
| Baseline | 普通 AI 对话，不加载本仓库 |
| Workflow | 加载 `PORTABLE.md` 或 `SKILL.md` 后运行 |

每个任务：

- 至少使用 2 个支持联网的 AI 平台；
- 每个平台、每组运行 3 次；
- 每次使用新对话；
- 输入保持一致；
- 记录完整来源；
- 运行后人工打开关键来源核验。

每个任务至少产生：

```text
2 个平台 × 2 个条件 × 3 次运行 = 12 份结果
````

## 三、评分

每项 0—2 分，总分 24 分。

| #  | 维度       | 0 分     | 1 分        | 2 分                 |
| -- | -------- | ------- | ---------- | ------------------- |
| 1  | 业务决策     | 未定义     | 模糊         | 明确且可执行              |
| 2  | 我方基线     | 缺失      | 不完整        | 用户、场景、能力、指标和约束完整    |
| 3  | 候选池      | 随意列举    | 有候选但无逻辑    | 候选类型和初筛理由明确         |
| 4  | 核心竞品筛选   | 仅凭知名度   | 有理由但无权重    | 加权筛选并解释排除           |
| 5  | Claim 证据 | 无来源     | 有来源但不可追踪   | Claim ID、来源、时间和等级完整 |
| 6  | 来源有效性    | 多数无效    | 少量无效       | 关键来源均可访问            |
| 7  | 主张匹配度    | 来源不支持主张 | 部分支持       | 关键来源直接支持主张          |
| 8  | 事实与推论    | 混合      | 偶尔区分       | 系统标注                |
| 9  | 时效性      | 动态信息无时间 | 部分有时间      | 动态信息均核验或标风险         |
| 10 | 机会判断     | 竞品有即建议做 | 简单评估       | 用户价值、适配、成本、风险完整     |
| 11 | 动作建议     | 空泛      | 有优先级但无执行信息 | 动作类型、优先级、指标和前置条件完整  |
| 12 | 无依据结论    | 多处      | 1—2 处      | 0 处                 |

## 四、附加指标

记录：

* 无效链接数量；
* 来源与主张不匹配数量；
* 过期动态事实数量；
* 编造功能、价格、评论或份额数量；
* 启动追问轮数；
* 候选池数量；
* 核心竞品数量；
* BUILD 建议数量；
* VALIDATE 建议数量；
* 三次运行核心判断一致度；
* 总完成时间；
* 是否生成我方基线；
* 是否生成项目状态；
* 是否生成证据账本。

## 五、测试任务

### Task 1：服务型公众号回访机制

```text
某运营商服务型微信公众号准备从纯办理查询转型为办理、内容和权益综合入口。请做竞品分析，目标是决定应该保留和优先验证哪些回访机制。
```

检查：

* 是否建立运营商自身基线；
* 是否区分公众号、App 和小程序；
* 是否把功能存在与提升回访分开；
* 是否把高潜力但低证据建议标为 VALIDATE。

### Task 2：AI 用户研究工具

```text
分析 AI 用户研究工具市场。目标是决定独立研究者产品应该优先切入哪个用户群，以及应该建设、验证、观察或拒绝哪些能力。
```

检查：

* 是否区分研究辅助、研究自动化和研究替代；
* 是否合理选择直接、间接和标杆对象；
* 是否核验价格和当前功能；
* 是否使用用户证据判断未满足需求。

### Task 3：简历岗位匹配产品

```text
分析 AI 简历岗位匹配产品。目标是决定先切入哪个垂直岗位市场，并确定第一版产品需要验证的差异化能力。
```

检查：

* 是否区分搜索、推荐、匹配、评估和自动投递产品；
* 是否建立团队资源和渠道约束；
* 是否避免把所有 AI 招聘工具放入核心竞品；
* 是否给出 P0 VALIDATE 方案。

## 六、来源核验

每份结果随机抽取至少 5 个关键 Claim，人工检查：

| Claim ID | 来源可访问 | 来源支持主张 | 时间有效 | 等级合理 |
| -------- | :---: | :----: | :--: | :--: |
|          |       |        |      |      |

以下任一情况记为严重错误：

* 来源不存在；
* 来源内容与引用不符；
* 把搜索摘要当原文；
* 编造产品功能；
* 编造价格；
* 编造用户评论；
* 把官方宣传写成实际效果。

## 八、通过标准

只有满足以下条件，才可在 README 中写“经 Benchmark 验证”：

1. Workflow 平均分比 Baseline 高至少 20%；
2. Workflow 编造来源、功能、价格、评论和市场数据数量为 0；
3. 关键来源有效率至少 95%；
4. 来源与主张匹配率至少 90%；
5. Workflow 三次运行的核心建议方向具有可解释的一致性；
6. 两个平台均表现改善；
7. 至少 80% 的 Workflow 输出可直接支持业务讨论；
8. 高风险 BUILD 建议不得仅由 C/D 级证据支持。
